### Вопрос 50 (MUST)
**В чём разница между двумя импортами?**

```typescript
import { test, expect } from "@playwright/test";
import { LoginPage } from "./pages/LoginPage";
```

- **`"@playwright/test"`** — импорт из **внешней библиотеки**. Это пакет который лежит в `node_modules/`. Говорим: "возьми `test` и `expect` из установленной библиотеки Playwright."

- **`"./pages/LoginPage"`** — импорт из **своего файла**. `./` означает "текущая папка". Говорим: "возьми `LoginPage` из моего собственного файла."

Ключевая разница:
- `"@..."` = внешняя библиотека (из `node_modules/`)
- `"./"` = твой собственный файл

---

### Вопрос 51 (MUST)
**Что такое `node_modules/` и стоит ли её редактировать?**

`node_modules/` — папка где хранятся **сами файлы всех установленных библиотек** (включая Playwright). Создаётся автоматически командой `npm install`.

**Редактировать никогда не нужно** потому что:
- Создаётся автоматически
- Очень большая папка
- Любые изменения перезапишутся при следующей установке

Поэтому её добавляют в `.gitignore` — не нужно отправлять на GitHub.

---

### Вопрос 52 (OPTIONAL)
**Что содержит `package.json`?**

`package.json` — "паспорт" проекта. Содержит:
- **Название** проекта
- **Версия** проекта
- **Список библиотек** (зависимостей)
- **Скрипты** — например команда `npm test`

### Вопрос 53 (PRACTICE)
**В коде ниже определи: импорты, async функцию, деструктуризацию, локатор, действие, проверку.**

```typescript
import { test, expect } from "@playwright/test";

test.describe("Login page", () => {
  test.beforeEach(async ({ page }) => {
    await page.goto("https://example.com/login");
  });

  test("should show error for wrong password", async ({ page }) => {
    await page.getByPlaceholder("Email").fill("user@test.com");
    await page.getByPlaceholder("Password").fill("wrongpassword");
    await page.getByRole("button", { name: "Login" }).click();
    await expect(page.getByText("Invalid credentials")).toBeVisible();
  });
});
```

- **Импорт:** `import { test, expect } from "@playwright/test"`
- **Async функции:** `async ({ page }) =>` — есть в `beforeEach` и в самом тесте
- **Деструктуризация:** `{ page }` — фигурные скобки вокруг page в обоих местах
- **Локаторы:** `getByPlaceholder("Email")`, `getByPlaceholder("Password")`, `getByRole("button", { name: "Login" })`, `getByText("Invalid credentials")`
- **Действия:** `.fill()`, `.click()`
- **Проверка:** `expect(...).toBeVisible()`

> ⚠️ Деструктуризация — это `{ page }` внутри `async ({ page })`. Её легко не заметить — она есть в КАЖДОЙ async функции теста!

### Вопрос 54 (PRACTICE)
**Что не так в этом коде?**

```typescript
function getTimeout(seconds: number): string {
  return seconds * 1000;
}
```

**Ошибка:** возвращаемый тип указан как `: string`, но функция умножает число на 1000 — результат число, а не текст.

**Исправление:**
```typescript
function getTimeout(seconds: number): number {
  return seconds * 1000;
}
```

### Вопрос 55 (PRACTICE)
**Что не так в этом коде?**

```typescript
const config = { baseURL: "https://staging.example.com" };
console.log(config.baseUrl);
```

### Вопрос 56 (MUST)
**Порядок локаторов от лучшего к худшему:**

1. `getByRole` — самый предпочтительный
2. `getByLabel`
3. `getByPlaceholder`
4. `getByText`
5. `getByTestId`
6. CSS селекторы (`.locator()`)
7. XPath — наименее предпочтительный

> ⚠️ Частая ошибка: путать порядок `getByText` и `getByTestId`. Запомни: `getByText` идёт **перед** `getByTestId`.

---

### Вопрос 57 (MUST)
**Почему XPath стоит последним? Когда его можно использовать?**

XPath привязан к **структуре HTML** — например `//div[3]/button` означает "третий div, внутри него кнопка". Если разработчик добавит ещё один div — XPath сломается.

**Когда можно использовать:** только когда вообще нет другого варианта — нет `data-testid`, нет роли, нет текста, нет placeholder. Это крайний случай. Лучше сначала попросить разработчика добавить `data-testid`.

---

### Вопрос 58 (PRACTICE)
**Напиши `getByRole` локатор для кнопки "Login", заголовка "Installation", ссылки "Sign up".**

```typescript
page.getByRole('button', { name: 'Login' })
page.getByRole('heading', { name: 'Installation' })
page.getByRole('link', { name: 'Sign up' })
```

> ⚠️ Частые ошибки:
> - Роли пишутся с **маленькой буквы**: `'button'`, не `'Button'`
> - Заголовок = `'heading'`, не `'title'` и не `'Title'`
> - Ссылка = `'link'`, не `'Link'`
> - Внутри всегда `{ name: '...' }` — не просто `{'...'}`

---

### Вопрос 59 (PRACTICE)
**Что не так с каждым локатором?**

```typescript
page.locator("//div[3]/button")
```
❌ XPath — зависит от структуры HTML. Добавят новый div — сломается.

```typescript
page.locator(".btn-xyz123")
```
❌ Автогенерированный CSS класс — в следующем билде сгенерируется другой.

```typescript
page.locator("div > div > form > button")
```
❌ Слишком длинная цепочка CSS — любое изменение вёрстки сломает.

```typescript
page.locator(".item").nth(5)
```
❌ Порядковый номер — если добавят новые элементы, найдёт не тот.

**Все четыре плохи потому что нестабильны и зависят от структуры HTML.** Правильнее использовать `getByRole` или `getByTestId`.

---

### Вопрос 60 (MUST)
**Разница между локаторами:**

- **`getByRole`** — находит по роли элемента: `'button'`, `'link'`, `'heading'`, `'textbox'`
```typescript
page.getByRole('button', { name: 'Login' })
```

- **`getByLabel`** — находит поле по его подписи (label рядом с полем)
```typescript
page.getByLabel('Email')
```

- **`getByPlaceholder`** — находит поле по серому тексту-подсказке внутри поля
```typescript
page.getByPlaceholder('Username')
```

- **`getByText`** — находит элемент по видимому тексту на странице
```typescript
page.getByText('Products')
```

- **`getByTestId`** — находит по атрибуту `data-testid` который разработчик добавил специально для тестов
```typescript
page.getByTestId('login-button')
```

## WEEK 3 — Действия, состояния, assertions

---

### Вопрос 61 (PRACTICE)
**Что такое `baseURL` и почему `page.goto("/")` лучше?**

`baseURL` — ключ в `playwright.config.ts` который хранит базовый адрес сайта.

```typescript
baseURL: 'https://www.saucedemo.com'
```

Когда `baseURL` задан — в тестах можно писать просто:
```typescript
await page.goto('/') // вместо 'https://www.saucedemo.com'
await page.goto('/inventory') // вместо 'https://www.saucedemo.com/inventory'
```

**Почему лучше:**
1. Быстрее писать
2. Если адрес сайта поменяется — меняешь **одну строку** в конфиге, а не в каждом тесте

---

### Вопрос 62 (MUST)
**3 главных состояния элемента:**

1. **Visible / Hidden** — виден / скрыт на странице
   - Пример: модальное окно с ошибкой появляется только после неудачного логина

2. **Enabled / Disabled** — активен / заблокирован
   - Пример: кнопка Submit заблокирована пока не заполнены все поля

3. **Attached / Detached** — присутствует в HTML / отсутствует
   - Пример: список товаров появляется только после загрузки данных

> ⚠️ Запомни пары: Visible/**Hidden** (не Invisible), Enabled/**Disabled**, Attached/**Detached**

---

### Вопрос 63 (MUST)
**Что такое auto-waiting?**

Перед каждым действием Playwright **автоматически** проверяет что элемент:
- **Visible** — виден на странице
- **Enabled** — не заблокирован
- **Stable** — не двигается (анимация закончилась)
- **Attached** — присутствует в DOM
- **Receives events** — может принять клик (не перекрыт другим элементом)

Поэтому не нужно писать `page.waitForTimeout(3000)` перед каждым кликом — Playwright сам ждёт.

---

### Вопрос 64 (MUST)
**Что Playwright НЕ ждёт автоматически?**

1. **Сетевые запросы (API)** — страница может выглядеть загруженной, но данные с сервера ещё не пришли
   → Использовать: `await page.waitForLoadState('networkidle')`

2. **Появление нового элемента после действия** — кликнул на кнопку, появилось модальное окно
   → Использовать: `await expect(locator).toBeVisible()`

3. **Смена URL после действия** — после логина страница редиректит на `/inventory`
   → Использовать: `await expect(page).toHaveURL(/inventory/)`

```typescript
await page.goto('/')                          // открыть страницу
```

### Вопрос 66 (PRACTICE)
**Напиши assertions:**

```typescript
// URL содержит /inventory
await expect(page).toHaveURL(/inventory/)

// Элемент с текстом "Products" виден
await expect(page.getByText('Products')).toBeVisible()

// Корзина показывает ровно "1"
await expect(page.locator('.shopping_cart_badge')).toHaveText('1')

// Ошибка содержит подстроку "Username is required"
await expect(page.getByText('Username is required')).toContainText('Username is required')

// Кнопка Login активна
await expect(page.getByRole('button', { name: 'Login' })).toBeEnabled()

// На странице ровно 6 товаров
await expect(page.locator('.inventory_item')).toHaveCount(6)
```

> ⚠️ Частые ошибки:
> - `toBeVisible()` — без аргументов!
> - `toBeEnabled()` — без аргументов!
> - `toHaveCount(6)` — число **без кавычек**, не `'6'`
> - Не забывать `await expect(...)` — полная строка, не только метод

### Вопрос 67 (MUST)
**Что такое failure message и зачем добавлять?**

Failure message — кастомное сообщение которое появляется когда assertion падает.

```typescript
await expect(page.locator('.cart')).toHaveText('1', {
  message: 'Cart badge should show 1 item after adding product'
});
```

**Зачем:** когда тест падает в CI/CD ночью — ты открываешь отчёт утром и сразу видишь что пошло не так, без расследования.

---

### Вопрос 68 (MUST)
**Разница между `toHaveText` и `toContainText`?**

- **`toHaveText("1")`** — текст элемента должен **точно равняться** "1"
- **`toContainText("1")`** — текст элемента должен **содержать** "1" (могут быть другие символы)

```typescript
// Элемент содержит текст: "1 item in cart"
await expect(locator).toHaveText("1")              // ❌ упадёт
await expect(locator).toHaveText("1 item in cart") // ✅ пройдёт
await expect(locator).toContainText("1")           // ✅ пройдёт
```

**Когда использовать:**
- `toHaveText` — когда знаешь точный текст (значок корзины = `"1"`)
- `toContainText` — когда важна только часть текста (сообщение об ошибке содержит "Username")

### Вопрос 69 (MUST)
**Почему Pick Locator — только отправная точка? 6-шаговый процесс валидации.**

**Почему только отправная точка:** Pick Locator предлагает локатор автоматически, но не всегда выбирает лучший — может предложить нестабильный CSS или XPath. Нужно проверить вручную.

**6 шагов валидации:**
1. Открой **DevTools** (правая кнопка → Inspect)
2. Найди нужный **элемент** в панели Elements
3. Посмотри **атрибуты** — есть ли `data-testid`, `role`, `placeholder`, `aria-label`?
4. Проверь **уникальность** через поиск в Elements (Ctrl+F) или через Console: `$$('локатор')` — должен найти **ровно 1 элемент**
5. Выбери **лучший локатор** по приоритету (getByRole → getByLabel → ...)
6. **Проверь в Playwright** — запусти тест и убедись что работает

---

### Вопрос 70 (MUST)
**4 стабильных атрибута в DevTools:**

1. **`data-testid`** — специально для тестов, самый стабильный
2. **`role`** — роль элемента (button, textbox, link)
3. **`placeholder`** — текст-подсказка внутри поля
4. **`aria-label`** — метка доступности

**Нестабильные** (избегать): автогенерированные классы `.btn-xyz123`, порядковые номера `div[3]`

### Вопрос 71 (PRACTICE)
**Как проверить уникальность локатора? 2 способа.**

**Способ 1 — через Elements:**
1. Открой DevTools → Elements
2. Нажми **Ctrl+F** (поиск)
3. Вставь локатор — должно найтись **ровно 1 совпадение**

**Способ 2 — через Console:**
1. Открой DevTools → Console
2. Напиши: `$$('[data-test="username"]')`
3. Нажми Enter
4. Должен вернуться массив с **1 элементом**

Если найдено больше одного — локатор не уникальный, нужно искать лучший.

> Пример результата в Console: `[input#user-name]` — один элемент = локатор уникальный ✅

---

### Вопрос 72 (PRACTICE)
**Что делать если нет ни одного стабильного локатора?**

Правильный порядок действий:
1. Сначала ищешь стабильный атрибут (`role`, `placeholder`, `data-testid`, `aria-label`)
2. Если ничего нет — **просишь разработчика добавить `data-testid`**. Это нормальная практика в командах.
3. Только если совсем никак — используешь XPath как **временное** решение

> ⚠️ XPath — не первый вариант! Сначала всегда просим разработчика добавить `data-testid`.

## WEEK 3 — Анти-паттерны и отладка

### Вопрос 73 (MUST)
**Назови 4 анти-паттерна которые курс говорит НЕ делать.**

1. **Жёсткие таймауты** — `page.waitForTimeout(5000)`. Плохо в обе стороны: слишком короткое — тест упадёт, слишком длинное — тесты медленные. Использовать вместо этого `await expect(locator).toBeVisible()`.

2. **Нестабильные локаторы** — XPath и автогенерированные CSS классы. Зависят от структуры HTML и ломаются при изменении вёрстки.

3. **Созависимые тесты** — Test B работает только если Test A прошёл. Каждый тест должен быть **независимым** и работать сам по себе.

4. **Плохие названия тестов** — `"click button"`, `"fill form"`. Название должно описывать **поведение**, а не действие. Формула: начинать с `"should..."`.

---

### Вопрос 74 (MUST)
**Почему `page.waitForTimeout(5000)` плохо в обоих направлениях?**

- **Слишком короткое** (например 1000ms) — страница не успела загрузиться, тест пытается найти элемент которого ещё нет → тест падает.
- **Слишком длинное** (например 10000ms) — тест висит и ждёт даже когда элемент уже давно появился → тесты медленные, CI/CD занимает часы.

**Что использовать вместо:**
```typescript
await expect(locator).toBeVisible() // Playwright сам ждёт ровно столько сколько нужно
```

Playwright будет ждать пока элемент появится — не дольше и не короче. Умно! 🙂

```typescript
await expect(page).toHaveURL(/inventory/)
```

> `/inventory/` — это regex (регулярное выражение), проверяет что URL **содержит** слово "inventory".

**Для негативного теста логина — локатор и assertion для сообщения об ошибке.**

> - Атрибут в кавычках: `[data-test=` **`"error"`** `]` — с кавычками внутри!

### Вопрос 77 (PRACTICE)


### Вопрос 78 (PRACTICE)
**После удаления товара — assertion что корзина пуста.**

```typescript
await expect(page.locator('.shopping_cart_badge')).toBeHidden()
```

> ⚠️ Метод называется `toBeHidden()`, не `toBeInvisible()` — такого метода не существует!
> Когда корзина пуста — значок исчезает с экрана, поэтому проверяем что он скрыт.

### Вопрос 79 (MUST)
**Что такое "Task 6 — Stability challenge"? Команда?**

**Task 6** — задание где нужно запустить тест **несколько раз подряд** чтобы убедиться в его стабильности. Если тест иногда проходит а иногда падает — это **flaky test** (нестабильный тест). Так не должно быть.

**Команда:**
```bash
npx playwright test --repeat-each=3
```

Запускает каждый тест 3 раза подряд. Все 3 раза зелёный = тест стабильный ✅

|----------------|-----------------|

### Вопрос 81 (MUST)
**4 самых частых типа падений тестов:**

1. **Timeout** — элемент не появился за отведённое время.
   Причина: неправильный локатор, страница не загрузилась, элемент скрыт.

2. **Element not found** — элемент вообще не существует на странице.
   Причина: неправильный локатор, страница не та, элемент ещё не появился.

3. **Strict mode violation** — локатор нашёл несколько элементов вместо одного.
   Причина: локатор не уникальный — например `getByText('Login')` нашёл 2 элемента.

4. **Assertion failed** — проверка не прошла.
   Причина: ожидали одно значение, получили другое. Например: ожидали `toHaveText('1')`, а в корзине `'2'`.

### Вопрос 82 (PRACTICE)
**Дана ошибка — что проверишь первым делом?**

```
Error: Timed out 5000ms waiting for expect(locator).toBeVisible()
Locator: getByText("Welcome back")
Received: <element not found>
```

Чеклист что проверить:
1. **Правильный ли текст в локаторе** — может на странице "Welcome Back" (с большой B) или "Welcome, back!"? TypeScript чувствителен к регистру.
2. **Есть ли элемент на странице вообще** — открой браузер и проверь глазами.
3. **Появляется ли элемент позже** — может страница ещё не загрузилась?
4. **Правильная ли страница** — может тест попал не туда?

> Первое что проверяем — **локатор и текст элемента**.

### Вопрос 83 (MUST)
**Сопоставь инструмент с ситуацией:**

- **`--ui`** — открывает UI Mode. Специальное окно где визуально видно как проходит каждый шаг теста. Лучший инструмент для ежедневной разработки.

- **`--headed`** — открывает обычный браузер где запускается тест. Видишь как Playwright кликает и заполняет поля. Проще чем UI Mode, но без дополнительных инструментов.

- **`--debug`** — открывает Playwright Inspector. Позволяет запускать тест **пошагово** — строчка за строчкой. Удобно когда нужно точно найти на какой строке всё ломается.

- **`--trace on`** — включает запись трейсов для всех тестов. После запуска можно открыть Trace Viewer и посмотреть детальную запись каждого шага.

- **`page.pause()`** — вставляешь прямо в код теста. Когда Playwright доходит до этой строки — тест **останавливается**. В этот момент можно открыть браузер и проверить состояние страницы.

> ⚠️ Разница между `--debug` и `page.pause()`:
> - `--debug` = пошаговый режим **с самого начала**
> - `page.pause()` = остановка в **конкретном месте** которое ты выбрала в коде

### Вопрос 84 (MUST)
**Что показывает Trace Viewer чего нет в HTML отчёте?**

**HTML отчёт** (`npx playwright show-report`):
- Какие тесты прошли ✅ и упали ❌
- Сколько времени занял каждый тест
- Общая картина по всем тестам

**Trace Viewer** показывает намного больше:
- **Скриншот каждого шага** — видишь как выглядела страница в каждый момент
- **Сетевые запросы** — какие запросы отправлялись на сервер
- **Консольные логи** — что писалось в консоль браузера
- **Временную шкалу** — когда именно что произошло
- **До и после каждого действия** — как страница выглядела до клика и после

Аналогия: HTML отчёт — табель с оценками. Trace Viewer — видеозапись всего урока с возможностью перемотать в любой момент. 🙂

**Тест падает в CI, но проходит локально. Есть `trace.zip`. Что проверишь первым в Trace Viewer — 3 вещи?**

### Вопрос 87 (PRACTICE)
**5-шаговый процесс отладки который рекомендует курс:**

### Вопрос 88 (MUST)
**Что такое "Strict mode violation" и как исправить?**

**Strict mode violation** — ошибка которая возникает когда локатор находит **несколько элементов** вместо одного. Playwright требует чтобы каждый локатор находил ровно 1 элемент.

Пример ошибки:
```
Error: strict mode violation: getByText('Login') resolved to 2 elements
```

**Как исправить:**
1. Открой DevTools → Console
2. Проверь локатор через `$$('локатор')`
3. Найди более уникальный атрибут (`data-testid`, `role` с конкретным `name`)
4. Замени локатор на уникальный

```typescript
// Было — находит 2 элемента
page.getByText('Login')

// Стало — уникально
page.getByRole('button', { name: 'Login' })
```

### Вопрос 89 (MUST)
**Типичный фикс для "Works locally, fails in CI"?**

**4 частые причины и решения:**

1. **Разная скорость интернета/сервера** — в CI медленнее, элемент не успевает загрузиться
   → Убрать жёсткие таймауты, использовать `expect(locator).toBeVisible()`

2. **Зависимые тесты** — порядок запуска в CI может отличаться от локального
   → Сделать каждый тест независимым

3. **Разные размеры экрана (viewport)** — в CI другой размер
   → Задать конкретный viewport в `playwright.config.ts`

4. **Не установлены браузеры в CI** — `node_modules` есть, а браузеров нет
   → Добавить `npx playwright install` в CI pipeline

### Вопрос 90 (OPTIONAL)
**Что обычно означает "Navigation interrupted"?**

Означает что одна навигация (переход на страницу) была **прервана другой навигацией** до того как первая завершилась.

> ⚠️ Не "два процесса одновременно" — а именно один **прерывает** другой!

**Типичные причины:**
1. Страница делает автоматический **редирект** пока ты ждёшь загрузки другой страницы
2. Клик вызывает переход, но предыдущая навигация ещё не завершилась

**Как исправить:**
```typescript
await page.waitForLoadState('networkidle')
// или
await expect(page).toHaveURL(/новый-урл/)
```

## WEEK 4 — Структура тестов и независимость

### Вопрос 91 (MUST)
**Сопоставь с назначением: `test.describe`, `test.beforeEach`, `test.afterEach`, `test.skip`, `test.only`**

- **`test.describe`** — группирует несколько тестов вместе под одним названием
```typescript
test.describe('Login tests', () => { ... })
```

- **`test.beforeEach`** — код который выполняется **перед каждым** тестом в группе (например логин)
```typescript
test.beforeEach(async ({ page }) => { await page.goto('/login'); })
```

- **`test.afterEach`** — код который выполняется **после каждого** теста (например уборка данных)

- **`test.skip`** — пропускает тест, не запускает его (временно, пока тест сломан)

- **`test.only`** — запускает **только этот** тест, игнорируя остальные (для отладки)

> ⚠️ Это методы внутри **кода теста**, не команды терминала!

---

### Вопрос 92 (MUST)
**Почему `test.only` опасен в Pull Request?**

Если разработчик **забыл убрать** `test.only` перед созданием PR — в CI/CD запустится **только этот один тест**, а все остальные будут **пропущены** (даже не проверены).

CI покажет зелёную галочку "1 passed" — но реально проверился только 1 тест из 50 (например). Остальные могли быть сломаны, а никто не узнает.

**Поэтому:** перед PR всегда проверять что нигде не осталось `test.only` — это пункт в PR self-review checklist.

### Вопрос 93 (PRACTICE)
**Перепиши используя `beforeEach`:**

```typescript
test("test A", async ({ page }) => {
  await page.goto("/login");
  await page.getByPlaceholder("Username").fill("standard_user");
  // ... rest of test A
});

test("test B", async ({ page }) => {
  await page.goto("/login");
  await page.getByPlaceholder("Username").fill("standard_user");
  // ... rest of test B
});
```

**Ответ:**
```typescript
test.describe('Login', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto("/login");
    await page.getByPlaceholder("Username").fill("standard_user");
  });

  test('Test A', async ({ page }) => {
    // rest of test A
  });

  test('Test B', async ({ page }) => {
    // rest of test B
  });
});
```

> ⚠️ Частые синтаксические ошибки: не забывать фигурную скобку `{` после `async ({ page }) =>`, и правильно закрывать скобки `});` в конце каждого блока.

### Вопрос 94 (MUST)
**Что означает "test independence"? Почему плохо если Test B проходит только после Test A?**

**Test independence** = каждый тест должен сам себя подготовить и пройти **без оглядки на другие тесты**.

**Почему плохо если есть зависимость:**
1. Если Test A упадёт — автоматически упадёт и Test B (хотя Test B сам по себе может быть рабочим)
2. **Нельзя запустить тесты по отдельности** — например для отладки нужен только Test B, а он не работает без Test A
3. Тесты должны работать **в любом порядке** — зависимость это нарушает

### Вопрос 95 (PRACTICE)
**Один тест добавляет товар в корзину, другой ожидает что товар уже там. Что не так?**

Второй тест **зависит** от первого — он не самодостаточный (нарушение test independence, вопрос 94).

**Проблема:** если первый тест упадёт или не запустится — второй тест тоже упадёт, хотя логика второго теста может быть полностью корректной.

**Решение:** второй тест должен **сам** добавить товар в корзину (например в своём `beforeEach` или в начале теста) перед тем как проверять что товар там есть.

### Вопрос 96 (PRACTICE)
**Broken Test #1 (неправильный локатор) — диагностический процесс:**

1. **Читаем ошибку** — Playwright скажет что элемент не найден или timeout
2. **Находим строку кода** где произошла ошибка
3. **Идём в DevTools** — открываем страницу, находим нужный элемент
4. **Проверяем атрибуты** через Console: `$$('локатор')` — должен найтись правильный уникальный элемент
5. **Сравниваем** с тем что написано в тесте — может опечатка (например `'Pasword'` вместо `'Password'`)
6. **Исправляем** локатор на правильный

### Вопрос 97 (PRACTICE)
**Что требует шаблон отчёта "Root cause / Fix / How I verified"?**

- **Root cause** (корневая причина) — что именно было сломано и почему. Например: "Placeholder написан с опечаткой 'Pasword' вместо 'Password'"
- **Fix** (исправление) — что именно ты изменила. Например: "Исправила локатор на правильный placeholder"
- **How I verified** (как проверила) — как убедилась что исправление работает. Например: "Запустила тест — прошёл успешно, дополнительно прогнала через `--repeat-each=3`"

**Зачем:** используется в Pull Request — коллега сразу видит что было не так, что исправлено и откуда уверенность что работает, без необходимости разбираться в коде самому.

### Вопрос 98 (MUST)
**Три стадии файла: working tree → staged → committed. Команды между стадиями?**

- **Working tree** — ты изменила файл на компьютере, Git видит изменение, но ничего ещё не "взял" в работу
- **Staged** — командой `git add имя_файла` (или `git add .`) говоришь Git: "включи эти изменения в следующий коммит"
- **Committed** — командой `git commit -m "сообщение"` изменения **сохраняются навсегда** в истории проекта

Аналогия: working tree = письмо на столе. staged = письмо в конверте. committed = конверт заклеен и подписан.

> ⚠️ Это всё ещё **локально**! Чтобы попасть на GitHub нужен ещё `git push` — отдельный, четвёртый шаг, не входит в эти три стадии.

---

### Вопрос 99 (MUST)
**Разница между локальным репозиторием и удалённым (remote)?**

**Репозиторий** — хранилище проекта вместе со всей историей изменений (коммитов).

- **Локальный репозиторий** — копия проекта **на твоём компьютере**. Видишь только ты.
- **Удалённый (remote) репозиторий** — копия проекта **на сервере** (например GitHub). Видна всей команде.

Команды:
- `git push` — отправляет изменения из локального репозитория в удалённый
- `git pull` — скачивает изменения из удалённого репозитория к тебе

---

### Вопрос 100 (MUST)
**Почему курс говорит установить `git config user.email` перед первым коммитом?**

Если email не настроен или не совпадает с GitHub:
- Коммиты технически пройдут, но **не будут привязаны** к твоему GitHub аккаунту
- Не будет видно твоего имени, аватарки, email рядом с коммитом
- Ни ты, ни команда не поймёте кто сделал изменения

Поэтому перед **первым** коммитом обязательно настроить:
```bash
git config user.email "твой_email@example.com"
git config user.name "Твоё Имя"
```
