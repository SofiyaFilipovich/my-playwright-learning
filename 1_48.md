### Вопрос 1 (MUST)
**Назови 5 действий в терминале из Week 0 и команду для каждого.**

1. **Открыть терминал** — это действие, не команда. На Mac: Cmd + пробел → написать "Terminal" → Enter
2. **Создать папку** — `mkdir my-playwright-learning` (mkdir = make directory)
3. **Перейти в папку** — `cd my-playwright-learning` (cd = change directory)
4. **Посмотреть содержимое папки** — `ls` (ls = list)
5. **Выполнить команду / проверить версию Node.js** — `node --version`
6. **Открыть VS Code** — `code .` (точка = текущая папка)

### Вопрос 2 (OPTIONAL)
**Почему Playwright требует определённую версию Node.js? Что делать если видишь v14?**

Playwright — современная технология, которая использует последние возможности JavaScript. Старые версии Node.js эти возможности не поддерживают, поэтому Playwright на них просто не запустится.

- Минимальная версия: **Node.js 18**
- Рекомендуемая версия: **Node.js 22**
- Если видишь v14 — нужно обновить. Проще всего: скачать новую версию с сайта **nodejs.org**

**Красное волнистое подчёркивание под `page.fil("text")`. Что это и как исправить?**

### Вопрос 16 (PRACTICE)
**Show browser и Show trace viewer — что делают? Когда включать/выключать?**

Находятся в настройках VS Code: шестерёнка → Settings → поиск "playwright"

**Show browser (Playwright: Reuse Browser):**
- Включён → при запуске теста открывается браузер и ты видишь всё вживую
- Включать: когда пишешь новый тест или ищешь ошибку в конкретном тесте
- Выключать: когда запускаешь все тесты подряд — так быстрее

**Show trace viewer (Playwright: Show Trace):**
- Включён → после запуска тестов автоматически открывается Trace Viewer с детальной записью каждого шага
- Включать: когда тест упал и хочешь разобраться почему
- Выключать: в обычной работе — чтобы не открывался лишний экран после каждого теста

> ⚠️ Trace Viewer — это НЕ терминал. Это отдельное визуальное окно с записью шагов теста (скриншоты каждого действия).

### Вопрос 17 (MUST)
**M, U, D в Source Control — что означает каждая буква?**

Source Control находится в Activity Bar — иконка с разветвлением 🔀 (третья сверху).

- **M** (Modified) — файл был **изменён**. Ты открыла существующий файл и что-то поправила.
- **U** (Untracked) — файл **новый**, Git его ещё не знает. Ты создала новый файл.
- **D** (Deleted) — файл был **удалён**.

Аналогия: M = сотрудник поменял фамилию, U = новый сотрудник которого ещё не внесли в базу, D = сотрудник уволился.

## WEEK 1 — Первый Playwright проект

---

### Вопрос 18 (MUST)
**Что такое Playwright и зачем используется?**

Playwright — инструмент для автоматизации тестирования браузерных приложений. Написан на JavaScript/TypeScript командой Microsoft.

Вместо того чтобы каждый раз вручную проходить одни и те же тест-кейсы — пишешь код, который делает это автоматически. Экономит время и человеческие ресурсы.

---

### Вопрос 19 (MUST)
**Какая команда создаёт новый Playwright проект с нуля?**

```bash
npm init playwright@latest
```

Эту команду запускаешь в терминале из папки проекта. Она создаёт все нужные файлы и устанавливает браузеры.

> ⚠️ Это команда для **терминала**, не для редактора!

---

### Вопрос 20 (PRACTICE)
**Файлы и папки после `npm init playwright@latest` — что каждый делает?**

- **`tests/`** — папка где хранятся все твои тесты (`.spec.ts` файлы)
- **`playwright.config.ts`** — главный файл настроек Playwright (браузеры, baseURL, trace и т.д.)
- **`package.json`** — "паспорт" проекта: название, версия, список используемых библиотек
- **`package-lock.json`** — точные версии всех установленных библиотек. Создаётся автоматически, **руками не трогаем**
- **`.gitignore`** — список файлов/папок которые Git должен игнорировать и не отправлять на GitHub
- **`node_modules/`** — сами файлы всех установленных библиотек (очень большая папка). Поэтому она в `.gitignore` — не нужно отправлять на GitHub

> ⚠️ Разница между `package-lock.json` и `node_modules/`:
> - `package-lock.json` = **список версий** библиотек
> - `node_modules/` = **сами файлы** библиотек

---

### Вопрос 21 (MUST)
**4 главных ключа в `playwright.config.ts`**

1. **`testDir`** — где искать тесты. У нас это папка `tests/`
2. **`baseURL`** — базовый адрес сайта. Например `https://www.saucedemo.com`. Чтобы в тестах писать `page.goto("/")` вместо полного URL каждый раз
3. **`trace`** — записывать ли трейсы. Например `'on'` = всегда, `'retain-on-failure'` = только при падении
4. **`projects`** — список браузеров для запуска тестов (chromium, firefox, webkit)

> ⚠️ Точные названия: `testDir`, `baseURL`, `trace`, `projects` — не путай написание!

**Что делают два теста в `example.spec.ts`?**

### Вопрос 29 (PRACTICE)
**Когда НЕ стоит автоматизировать тест?**

1. **Исследовательское тестирование** — творческий процесс поиска неожиданных багов. Автомат не умеет "исследовать".
2. **Одноразовый тест** — нужен один раз и больше никогда. Писать автоматизацию ради одного запуска — трата времени.
3. **Требования меняются очень часто** — тест будет постоянно ломаться, поддерживать дороже чем проверить руками.
4. **Сложные визуальные проверки** — "выглядит ли красиво" — компьютер не может оценить эстетику.

## WEEK 2 — TypeScript для QA

---

### Вопрос 31 (MUST)
**Что TypeScript добавляет к JavaScript?**

TypeScript добавляет **типы** (типизацию).

Ты указываешь какой формат данных ожидается в переменной — текст (`string`), число (`number`), true/false (`boolean`). Если передашь не тот формат — TypeScript покажет ошибку **до запуска кода**.

Аналогия: JavaScript — коробка куда можно положить что угодно. TypeScript — коробка с подписью "только яблоки". Положишь апельсин — сразу скажет "ошибка!".

---

### Вопрос 32 (MUST)
**Когда `const`, а когда `let`? Правило по умолчанию?**

- **`const`** — создаём переменную которая **не будет меняться**
- **`let`** — создаём переменную которая **будет меняться**

**Правило по умолчанию: всегда используй `const`.** Переходи на `let` только если точно знаешь что значение будет меняться.

---

### Вопрос 33 (MUST)
**Что такое type inference (вывод типов)? Пример.**

Type inference — TypeScript **сам догадывается** какой тип у переменной, без явного указания.

```typescript
const username = "standard_user"; // TypeScript сам понял: это string
const count = 5;                  // TypeScript сам понял: это number
```

Аналогия: показываешь яблоко — не говоришь что это. TypeScript сам видит "ага, это фрукт".

---

### Вопрос 34 (OPTIONAL)
**Когда писать явные аннотации типов?**

Когда TypeScript сам не может догадаться — например в **параметрах функций**:

```typescript
// TypeScript не знает что будет передано — пишем явно
function login(username: string, password: string) { ... }
```

### Вопрос 35 (PRACTICE)
**Перепиши используя template string:**
```typescript
const url = "https://" + env + ".example.com/login";
```

**Ответ:**
```typescript
const url = `https://${env}.example.com/login`;
```

**Правила template string:**
1. Используем **обратные кавычки** `` ` `` (backtick — кнопка слева от цифры 1), не обычные кавычки `"` или `'`
2. Переменную вставляем через **`${}`**
3. **Никаких лишних кавычек внутри!** `"` внутри backtick — ошибка
4. **Никаких пробелов** вокруг переменной: `${env}`, не `${ env }`

> ⚠️ Частая ошибка: писать `"https://" ${env}` с кавычками — неправильно!
> Правильно: `` `https://${env}` `` — только обратные кавычки снаружи.

### Вопрос 36 (MUST)
**Как получить первый элемент массива? Что такое массив?**

**Массив** — список элементов одного типа. Например: список пользователей, список товаров.

```typescript
const users = ["standard_user", "locked_out_user", "problem_user"];
```

**Как получить первый элемент:** через индекс в квадратных скобках.

> ⚠️ Важно: счёт начинается с **0**, не с 1!

```typescript
users[0] // "standard_user" — первый элемент
users[1] // "locked_out_user" — второй элемент
users[2] // "problem_user" — третий элемент
```

### Вопрос 37 (PRACTICE)
**Напиши typed объект `User` с полями `email`, `password` и опциональным `role`.**

```typescript
type User = {
  email: string;
  password: string;
  role?: string;
}
```

> ⚠️ Частые ошибки:
> - Писать `const User` вместо `type User` — неправильно! Для описания структуры используем `type`.
> - Писать `"string"` в кавычках — неправильно! Тип пишется **без кавычек**: `string`, не `"string"`.

### Вопрос 38 (MUST)
**Что означает `?` в `role?: string`?**

`?` означает что поле **опциональное** — оно может быть, а может не быть.

Объект `User` можно создать без поля `role` — TypeScript не будет ругаться.

### Вопрос 39 (MUST)
**Что делают фигурные скобки в `async ({ page }) => { ... }`?**

Это **деструктуризация**.

Playwright передаёт тесту большой объект со всем что может понадобиться: `page`, `browser`, `context` и т.д. Фигурные скобки говорят: "из всего этого объекта возьми мне **только** `page`."

```typescript
// Без деструктуризации — длинно
async (fixtures) => {
  const page = fixtures.page;

// С деструктуризацией — коротко
async ({ page }) => {
```

Аналогия: тебе привезли коробку с фруктами. Ты говоришь "дайте мне только яблоко" — берёшь только нужное.

---

### Вопрос 40 (OPTIONAL)
**Почему Playwright передаёт `page` внутри объекта, а не как обычный аргумент?**

Когда `page` внутри объекта — **порядок не важен**. Берёшь только то что нужно, в любом порядке:

```typescript
async ({ page, browser }) => { ... }
async ({ browser, page }) => { ... } // то же самое!
```

Если бы были обычные аргументы — порядок был бы строгим. Объект гибче.

### Вопрос 41 (PRACTICE)
**Напиши функцию `formatTestEmail(name: string): string`**

```typescript
function formatTestEmail(name: string): string {
  return `${name.toLowerCase()}@test.com`;
}
```

Разбор:
- `function formatTestEmail` — название функции
- `(name: string)` — принимает текст
- `: string` — возвращает текст
- `name.toLowerCase()` — делает "Alice" → "alice"
- `` `${name.toLowerCase()}@test.com` `` — собирает "alice@test.com"

> ⚠️ После `: string` нет знака `=` — он используется только для `type` и `const`, не для функций.

Вызов функции:
```typescript
formatTestEmail("Alice") // → "alice@test.com"
formatTestEmail("Sofia") // → "sofia@test.com"
```

Функция универсальная — работает с любым именем!

### Вопрос 42 (MUST)
**Что означает `: void` как тип возвращаемого значения?**

`: void` означает что функция **ничего не возвращает**.

```typescript
function logMessage(message: string): void {
  console.log(message); // просто печатает, ничего не возвращает
}
```

Типы которые может возвращать функция:
- `: string` — возвращает текст
- `: number` — возвращает число
- `: boolean` — возвращает true/false
- `: void` — ничего не возвращает

Аналогия: просишь коллегу отправить письмо — он отправляет, но ничего тебе не возвращает. Его "результат" это `void`.

### Вопрос 43 (OPTIONAL)
**Что делает оператор `?.` (optional chaining)?**

`?.` — если значение существует → возьми его, если нет → не падай с ошибкой, просто верни `undefined`.

```typescript
// Без ?. — может упасть с ошибкой если address не существует
const city = user.address.city;

// С ?. — безопасно: если address нет → вернёт undefined
const city = user.address?.city;
```

Аналогия: спрашиваешь "есть ли у тебя машина, и если есть — какого она цвета?" Если машины нет — просто говоришь "нет" вместо паники.

**Оператор** — это символ который что-то делает с данными (`+`, `=`, `?.`).

### Вопрос 44 (MUST)
**Что делает `await`?**

`await` говорит: "**подожди** пока эта операция завершится, и только потом выполняй следующую строку."

```typescript
// Без await — следующая строка не ждёт
page.click('button');        // клик ещё не завершился...
page.getByText('Success');   // ...а мы уже ищем текст! 💥

// С await — всё по порядку
await page.click('button');       // ждём пока клик завершится
await page.getByText('Success');  // только потом ищем текст ✅
```

---

### Вопрос 45 (MUST)
**Ошибка №1 у новичков в Playwright?**

**Забыть написать `await`.**

Это самая частая ошибка. Пишешь действие без `await` — следующая строка выполняется не дождавшись результата, и тест падает непонятным образом.

**Правило:** если видишь `page.` — почти всегда нужен `await` перед ним.

### Вопрос 46 (OPTIONAL)
**Что означает `Promise<void>`?**

`Promise<void>` = функция **асинхронная** И **ничего не возвращает**.

- `void` = ничего не возвращает
- `Promise` = асинхронная (выполняется не мгновенно, нужно ждать)

Когда пишешь `async function` — она автоматически возвращает `Promise`.

Аналогия: `void` = "принесу пустые руки", `Promise<void>` = "обещаю принести пустые руки, но чуть позже".

### Вопрос 47 (PRACTICE)
**Какие строки требуют `await`, а какие нет?**

```typescript
page.getByRole("button", { name: "Login" })              // ❌ await НЕ нужен
await page.getByRole("button", { name: "Login" }).click() // ✅ await нужен
await expect(page.getByText("Products")).toBeVisible()    // ✅ await нужен
```

**Правило:**
- Просто **локатор** (`getByRole`, `getByText`, `getByPlaceholder`) — `await` **не нужен**. Локатор только находит элемент, ничего не делает.
- **Действие** (`.click()`, `.fill()`, `.goto()`) — `await` **нужен**
- **Проверка** (`expect...toBeVisible()`) — `await` **нужен**

### Вопрос 48 (PRACTICE)
**Что означает `string | undefined` и как обработать?**

`string | undefined` = переменная может быть либо текстом, либо вообще ничем (`undefined`).

TypeScript не даст передать такую переменную в функцию которая ожидает только `string` — покажет ошибку.

**Как исправить — проверить что значение существует:**

```typescript
// Способ 1 — проверка через if
if (value !== undefined) {
  myFunction(value); // теперь TypeScript знает что это точно string
}

// Способ 2 — значение по умолчанию через ??
const safeValue = value ?? "default";
myFunction(safeValue);
```

Аналогия: тебе дали конверт — может быть там письмо, а может пусто. Прежде чем читать — проверяешь что конверт не пустой.
