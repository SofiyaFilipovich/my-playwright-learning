```bash

```

---

**Q4 (MUST): Почему git config user.email должен совпадать с email на GitHub?**

Потому что Git подписывает каждый коммит этим email. GitHub использует email для привязки коммитов к твоему аккаунту. Если они не совпадают — коммиты не будут отображаться в твоём профиле на GitHub.

---

**Q5 (MUST): Разница между gh auth login и gh auth setup-git?**

- `gh auth login` — авторизует GitHub CLI, подключает твой аккаунт GitHub к терминалу
- `gh auth setup-git` — настраивает Git чтобы он использовал GitHub CLI для аутентификации при push/pull

Нужны оба: первый авторизует, второй настраивает Git работать без паролей.

## НЕДЕЛЯ 1 — Первый проект

**Q18 (MUST): Что такое Playwright и зачем он используется?**

Playwright — это инструмент для автоматического тестирования веб-приложений в браузере, который выполняет действия (клики, заполнение форм, проверки) вместо ручного тестировщика.

---

**Q19 (MUST): Какая команда создаёт новый Playwright проект?**

```bash
npm init playwright@latest
```

---

**Q23 (MUST): Разница между запуском тестов через VS Code и через терминал?**

- **VS Code Testing panel** — визуальный интерфейс, удобно для одиночного запуска и дебаггинга
- **npx playwright test** — запуск из терминала, используется в CI/CD и для запуска всего набора тестов

Нужно уметь оба — VS Code для разработки, терминал для автоматизации.

---

**Q25 (MUST): Что открывает npx playwright test --ui?**

Открывает Playwright UI Mode — интерактивный интерфейс для запуска тестов с визуализацией шагов и скриншотами. Рекомендуется для ежедневной разработки тестов.

---

**Q26 (MUST): Что делает npx playwright show-report?**

Открывает HTML отчёт последнего запуска тестов в браузере — показывает какие тесты прошли, упали, сколько времени заняли.

---

**Q30 (MUST): Что такое E2E тест? Пример из SauceDemo.**

E2E (end-to-end) тест — тест который проверяет весь пользовательский сценарий от начала до конца, как реальный пользователь.

Пример: открыть saucedemo.com → ввести логин и пароль → нажать Login → проверить что открылась страница Products.

## НЕДЕЛЯ 2 — TypeScript

**Q31 (MUST): Что TypeScript добавляет к JavaScript?**

TypeScript добавляет статическую типизацию — возможность указывать типы данных (string, number, boolean), что помогает находить ошибки ещё до запуска кода.

---

**Q32 (MUST): Когда использовать const vs let?**

- `const` — когда значение не будет меняться (используй по умолчанию)
- `let` — только когда значение точно нужно менять

Правило: всегда используй `const`, переключайся на `let` только при необходимости.

---

**Q38 (MUST): Что означает ? в role?: string?**

Знак `?` означает что поле необязательное — оно может быть строкой или `undefined` (отсутствовать).

---

**Q39 (MUST): Что делают фигурные скобки в async ({ page }) => { ... }?**

Это деструктуризация — мы извлекаем поле `page` из объекта который передаёт Playwright. Вместо того чтобы писать `args.page`, мы сразу получаем `page`.

---

**Q44 (MUST): Что делает await?**

`await` говорит "подожди пока это действие завершится, и только потом иди дальше". Используется перед любым действием которое требует времени — открыть страницу, кликнуть, проверить элемент.

---

**Q45 (MUST): Самая частая ошибка новичков в Playwright?**

Забыть написать `await` перед действием. Без `await` код не дождётся выполнения и перейдёт дальше — тест будет нестабильным.

---

**Q51 (MUST): Что такое папка node_modules и можно ли её редактировать?**

`node_modules` — папка со всеми установленными библиотеками (Playwright и другими). Никогда не редактируй её вручную — она создаётся автоматически командой `npm install`.

## НЕДЕЛЯ 3 — UI тестирование

**Q56 (MUST): Порядок приоритета локаторов (от лучшего к худшему):**

1. `getByRole` — по роли элемента (button, link, heading)
2. `getByLabel` — по тексту label
3. `getByPlaceholder` — по placeholder поля ввода
4. `getByText` — по тексту элемента
5. `getByTestId` / `data-test` — по атрибуту тестирования
6. `locator('.css-class')` — по CSS классу
7. XPath — последний вариант, избегать

---

**Q60 (MUST): Разница между локаторами:**

- `getByRole` — для интерактивных элементов (кнопки, ссылки, заголовки)
- `getByTestId` — когда есть атрибут data-testid
- `getByLabel` — для полей формы с подписью
- `getByPlaceholder` — для полей ввода с placeholder
- `getByText` — для любого текста на странице

---

**Q68 (MUST): Разница между toHaveText и toContainText?**

- `toHaveText('1')` — элемент содержит **точно** этот текст
- `toContainText('1')` — элемент содержит этот текст **среди другого текста**

## НЕДЕЛЯ 4 — Дебаггинг и структура

**Q81 (MUST): 4 самых частых типа падений тестов:**

1. **Элемент не найден** — неверный локатор или элемент ещё не появился
2. **Timeout** — страница или элемент загружается слишком долго
3. **Wrong text/value** — проверяемое значение не совпадает с ожидаемым
4. **Strict mode violation** — локатор нашёл несколько элементов вместо одного

---

**Q84 (MUST): Что показывает Trace Viewer чего нет в HTML отчёте?**

Trace Viewer показывает пошаговую запись теста — скриншот на каждом шаге, сетевые запросы, консоль браузера. HTML отчёт показывает только итоговый результат.

---

**Q88 (MUST): Что такое Strict mode violation и как починить?**

Ошибка когда локатор находит несколько элементов вместо одного. Например, `getByRole('button', { name: 'Add to cart' })` нашёл 6 кнопок.

Починить: использовать более точный локатор — например `locator('[data-test="add-to-cart-sauce-labs-backpack"]')`.

---

**Q91 (MUST): Назначение test.describe, test.beforeEach, test.afterEach, test.skip, test.only:**

- `test.describe` — группирует связанные тесты
- `test.beforeEach` — выполняет код перед каждым тестом (например логин)
- `test.afterEach` — выполняет код после каждого теста (например очистка)
- `test.skip` — пропускает тест
- `test.only` — запускает только этот тест (осторожно — опасно в PR!)

---

**Q94 (MUST): Что такое независимость тестов?**

Каждый тест должен работать независимо от других — не зависеть от результатов предыдущих тестов. Если Test B проходит только после Test A, это плохой дизайн — Test A может упасть и Test B перестанет работать.

## КОНСПЕКТ — Команды и локаторы

### Команды терминала
```bash
npx playwright test                          # запустить все тесты
npx playwright test --project=chromium      # только в Chrome
npx playwright test saucedemo               # конкретный файл
npx playwright test --ui                    # UI Mode
npx playwright show-report                  # HTML отчёт
npx playwright show-trace trace.zip         # открыть трейс
```

### Локаторы
```typescript
page.getByPlaceholder('Username')
page.getByRole('button', { name: 'Login' })
page.getByText('Products')
page.locator('[data-test="add-to-cart-sauce-labs-backpack"]')
page.locator('.shopping_cart_badge')
page.locator('.shopping_cart_link')
```

### Действия
```typescript
.fill('текст')      // заполнить поле
.click()            // кликнуть
.goto('url')        // открыть страницу
```

### Проверки (expect)
```typescript
expect(page).toHaveTitle(/текст/)
expect(locator).toBeVisible()
expect(locator).not.toBeVisible()
expect(locator).toHaveText('текст')
expect(locator).toContainText('текст')
```
